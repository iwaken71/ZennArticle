## TL;DR

- Cursor のエージェントに「不要なブランチを整理して」と依頼したところ、**D ドライブ配下のデータが全て消失しました**(ゴミ箱にも残らず)
- Cursor のチャット履歴・ターミナル履歴も失われたため、**実行されたコマンドは特定できていません**
- AI に分析してもらった結果、推定される原因は `git clean -fdx` の誤爆、または削除コマンドのパス誤解決
- 教訓:**AI エージェントに破壊的コマンドの実行権限を渡すなら、Cursor 設定・OS・バックアップの 3 層で防御する**

同じ事故を起こす人を一人でも減らしたくて、ポストモーテムとして残します。

## 何が起きたか

2026 年 7 月 13 日の深夜 2 時頃、Cursor で作業中にエージェント(Agent / Fable, Composer)へこう依頼しました。

> 「不要なブランチを整理して」

Git ブランチの掃除のつもりでした。ブランチ削除(`git branch -d`)は作業ファイルに触らない操作なので、危険な依頼だとは思っていませんでした。

その後、別の作業を Agent に依頼していると、こんな返答が返ってきました。

> 「〇〇のデータがありません」
> 「緊急事態です、誰かに急速にデータが消されています」

何が起きているのか分からず混乱しました。作業中のプロジェクトがおかしくなっている……?

しかし気づいたときには、もっとひどいことが起きていました。**D ドライブ以下のデータが全て消えていました。**

- ゴミ箱には何もない(=エクスプローラー経由ではなくコマンドによる削除)
- Cursor のチャット履歴・ターミナル履歴も確認できず、実行されたコマンドを遡れない

作業していたプロジェクトだけでなく、他の作業フォルダや、D ドライブにインストールされていたアプリまで全てなくなっていました。
幸い、ほとんどのプロジェクトは GitHub で管理していたため致命傷は避けられましたが、push していない作業もあったので、かなりショックでした。

## 復旧のためにやったこと

まず、動いている Cursor の Agent を全て停止することから始めました。続けて次を実施しました。

1. **D ドライブへの書き込みを即座に停止**(削除直後ならファイル実体はディスクに残っていることが多く、上書きされる前が勝負)
2. クラウド同期(OneDrive / Google Drive 等)とそのゴミ箱・バージョン履歴の確認

一方で、次は今回やっていません。GitHub に push 済みのものが大半だったためです。

- エクスプローラーの「以前のバージョン」(シャドウコピー)の確認
- ファイル復元ソフトでのスキャン(Recuva / Windows File Recovery など。行う場合は**インストール先・復元先を必ず D ドライブ以外にする**)

結果として、GitHub に push していなかったローカル変更は戻りませんでした。

## ここから先は、AI に聞いた「原因」と「対策」です

チャット履歴もターミナル履歴も消えてしまったため、自力では原因を遡れません。そこで AI に事故の状況を伝え、「なぜ『ブランチ整理』の依頼で全消しが起こり得るのか」と「再発防止策」を分析してもらいました。以下はその回答を整理したものです。

## 原因の推定(なぜ「ブランチ整理」で全消しが起こるのか)

コマンドログが残っていないため断定はできませんが、AI エージェント事故の典型パターンから、可能性が高い順に整理します。

### 仮説 1: `git clean -fdx` の誤爆

エージェントは「整理して」という指示から、ブランチ削除に加えて「未追跡ファイルの掃除」まで気を利かせることがあります。そこで実行されがちなのが:

```bash
git clean -fdx
```

このコマンドは **Git が追跡していないファイル・フォルダを即時削除**します。ゴミ箱は経由しません。`-x` が付くと `.gitignore` されたファイルまで消します。

これが致命傷になる条件は 2 つ:

1. **カレントディレクトリが想定とズレていた**(リポジトリではなくドライブ直下で実行された)
2. **ドライブ直下やその近くに `.git` が存在していた**(過去の `git init` の残骸など)。この場合、Git から見た「リポジトリ」がドライブ全体になり、追跡外のデータが根こそぎ消えます

### 仮説 2: 削除コマンドのパス誤解決

「ブランチ用フォルダを消す」といった発想で、エージェントが

```powershell
Remove-Item -Recurse -Force $targetDir
```

のようなコマンドを組み立て、**変数が空・誤解決してドライブルートに向かう**パターン。シェルスクリプトの `rm -rf $DIR/` で `$DIR` が空だった、という古典的事故の AI エージェント版です。

### 仮説 3 ほか

拡張機能やクリーンアップ系ツールの誤動作、ドライブ側のファイルシステム破損なども理屈上はありえますが、「エージェントへの依頼直後」というタイミングを考えると、上記 2 つが本命です。

:::message
なお `git branch -D` 自体はワーキングツリーのファイルを消しません。犯人は「ブランチ整理のついでに実行された別の何か」のはずです。
:::

## 再発防止策

「Cursor 側の設定」「OS 側の防御」「バックアップ」の 3 層での対策が提案されました。1 層だけだと必ず穴が残ります。

### 1. Cursor 側:エージェントに破壊的操作をさせない

- **auto-run(YOLO モード)を無効化**し、コマンド実行は毎回手動 Approve に
- コマンドの **denylist** に破壊的コマンドを登録:
  - `rm` / `rmdir` / `del` / `Remove-Item` / `rd`
  - `git clean` / `git reset --hard` / `git worktree remove`
  - `format` / `mv`
- 特に `git clean` は「掃除」という響きのせいでエージェントが気軽に打ちがちなので必須
- ワークスペースは**プロジェクトフォルダ単位で開く**。ドライブ直下のような広い範囲を開かない(エージェントが触れる範囲=事故の最大範囲)

### 2. 指示の出し方を変える

「整理して」のような曖昧な依頼は、エージェントの「気の利かせ方」次第で破壊的操作に化けます。

**Before:**

> 不要なブランチを整理して

**After:**

> マージ済みのローカルブランチを一覧して。**削除はまだしないで。**

まず一覧(dry-run)だけ出させ、自分で確認してから「この 5 つを `git branch -d` で消して」と指示します。`-d`(小文字)ならマージ済みブランチしか消せないので二重に安全です。

削除・整理・クリーンアップ系のタスク全般で、**「まず対象一覧を出す → 人間が確認 → 実行」の 2 段階**を徹底します。

### 3. OS・環境側:そもそも消せない状態にする

- **重要データと開発作業領域を分離**。開発は `D:\dev` 配下に限定し、消えたら終わりのデータは開発ツールで開かない場所へ
- **リポジトリをドライブ直下に置かない**(`D:\.git` があると `git clean` の爆心地がドライブ全体になる)
- Windows の「**コントロールされたフォルダーアクセス**」(Windows セキュリティ → ランサムウェア防止)で、許可アプリ以外による重要フォルダの変更をブロック
- リポジトリ配下は「コミットしていないものはいつ消えてもいい場所」と割り切る(`git clean -fdx` は `.gitignore` 対象も消す)

### 4. バックアップ:消えても戻せる状態にする

- 重要フォルダの**クラウド同期を常時有効化**(バージョン履歴+クラウド側ゴミ箱が実質の保険)
- Windows の「**ファイル履歴**」を外付け HDD / NAS に対して有効化
- コードは**こまめに git push**(ローカル全滅でもリモートにあれば無傷)
- 月 1 回、**実際に復元できるかテスト**する。バックアップは「復元できて初めてバックアップ」

## 教訓まとめ

1. **AI エージェントへの「整理して」は破壊的操作の許可証になりうる。** 曖昧な指示ほどエージェントの解釈の幅が広がる
2. **`git clean` は削除コマンドである。** denylist に `rm` 系だけ入れて安心してはいけない
3. **コマンドによる削除はゴミ箱を経由しない。** 「ゴミ箱があるから大丈夫」は GUI 操作にしか通用しない
4. **エージェントが触れる範囲=事故の最大範囲。** ワークスペースのスコープを最小にする
5. **履歴も消えることがある。** 事故後の原因調査すらできなくなるので、防御は事前にしか張れない
6. **多層防御。** 設定・OS・バックアップのどれか 1 つに頼らない

## おわりに

AI エージェントにターミナルを渡すのは、便利さと引き換えに「人間なら躊躇するコマンドを、躊躇なく実行する存在」を招き入れることでもあります。エージェント自体を否定するつもりはまったくなく、これからも使い倒すつもりですが、**権限設計とバックアップだけは人間の仕事**です。

この記事が、誰かの D ドライブを救うことを願って。
